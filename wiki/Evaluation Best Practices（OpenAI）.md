# Evaluation Best Practices（OpenAI）

## 要約（約200字）

OpenAI公式の評価設計ガイド。生成AIの[[非決定性]]ゆえに従来のソフトウェアテストでは不十分とし、目的定義→[[評価データセット]]収集→指標定義→実行比較→[[継続評価]]の5段階プロセスを提示する。[[アーキテクチャ別の評価設計]]（単一ターン／ワークフロー／単一エージェント／マルチエージェント）で非決定性が入り込む箇所を特定して評価を置く考え方を示し、[[メトリクスベース評価]]・[[人間による評価]]・[[LLM-as-a-Judge]]の組み合わせ、[[エッジケース評価]]、[[Production Flywheel]]による継続改善までを体系化する。

## 主要概念

- [[評価駆動開発]]（eval-driven development）
- 評価設計5ステップ：目的 → [[評価データセット]] → 指標 → 実行比較 → [[継続評価]]
- [[アーキテクチャ別の評価設計]]
  - 単一ターン：指示遵守・機能的正しさ
  - ワークフロー：ステップごとの分離評価
  - 単一エージェント：[[ツール選択評価]]・引数の正確さ（Data precision）
  - マルチエージェント：[[エージェントハンドオフ評価]]
- 評価者の3類型
  - [[メトリクスベース評価]]（完全一致、ROUGE/BLEU、実行可能eval）
  - [[人間による評価]]（スコアカード、合議、Pass/Fail閾値）
  - [[LLM-as-a-Judge]]（ペア比較・単独採点・参照付き採点／位置バイアス・冗長性バイアスに注意）
- [[エッジケース評価]]（入力の多様性、文脈の複雑さ、ジェイルブレイク等）
- [[Production Flywheel]]（本番ログ→評価ケース化→改善の循環）
- アンチパターン：[[Vibeベース評価]]、[[正答率偏重]]（汎用メトリクス依存）、人間フィードバックとの較正欠如

## 他のソースとのつながり

- [[AIエージェント評価の極意（note・Anthropic記事解説）]] と評価者3類型（コード／モデル／人間）がほぼ一対一で対応する。OpenAI側は [[LLM-as-a-Judge]] のバイアス（位置・冗長性）と較正手順まで踏み込んでいる点がより詳細。
- [[AI Evals 評価設計（Arpable）]] は本ガイドのDataset / Grader / Harness / Flywheel整理と「evaluation best practices」を一次情報として引用している。
- 「非決定性が入る箇所に評価を置く」という [[アーキテクチャ別の評価設計]] の発想は本ガイド固有で、note記事の「エージェント種別ごとの評価」（用途別）とは切り口が異なる補完関係にある。
- 補足：Evalsプラットフォーム自体は2026年10月末に読み取り専用化、11月末に停止予定と明記されており、ツールではなく設計原則の部分に持続的価値がある。
